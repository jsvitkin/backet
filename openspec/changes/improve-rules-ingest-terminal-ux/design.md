## Context

`backet rules ingest` currently validates command input, calls the rules ingestion pipeline, waits for a `CommandResult`, and only then prints output. For large rulebooks, the pipeline can spend substantial time in page extraction, OCR fallback, file hashing, chunk storage, and FTS rebuild work with no visible terminal activity.

The current human completion output is also generated by the generic `emit_success()` renderer, which prints raw `CommandResult.data` fields. That is useful for JSON consumers, but poor for humans when values include long page arrays such as OCR pages and suspect pages.

The existing boundaries should remain intact:

- CLI owns terminal presentation.
- Rules ingestion owns PDF inspection, extraction, OCR fallback, chunking, storage, and audit metadata.
- The Obsidian vault owns `.backet/rules/` as durable per-vault state.
- Source PDFs remain external to the vault.
- Skills consume CLI results and should not implement progress display themselves.

## Goals / Non-Goals

**Goals:**

- Make a human `backet rules ingest` run visibly alive before long-running work begins.
- Show progress through the main ingestion phases, especially page extraction and OCR fallback.
- Keep JSON output deterministic and unpolluted by human progress/status text.
- Replace raw human dumps of long diagnostic arrays with bounded summaries, clear labels, and useful next commands.
- Preserve the existing rules corpus schema and retrieval behavior.

**Non-Goals:**

- No changes to source PDF handling, persisted chunk content, precedence behavior, query behavior, or audit semantics.
- No remote progress reporting, telemetry, remote OCR, or remote PDF processing.
- No new skill behavior.
- No normalized mechanics layer or rule-card extraction.

## Decisions

### 1. Add progress events from the ingestion pipeline, rendered by the CLI

The ingestion layer should expose a small callback/event stream while work proceeds. The CLI should pass a reporter in human mode and render those events with Rich. The rules layer should not import Rich or terminal UI concepts.

Conceptually:

```text
rules.py
  ingest_rulebook(..., progress=callback)
    progress(phase="inspect", message="Inspecting PDF")
    progress(phase="extract", current=34, total=431, counters={...})
    progress(phase="ocr", current=34, total=431, message="OCR fallback")
    progress(phase="store", current=600, total=1169)

cli.py
  human mode -> Rich progress/status on stderr
  json mode  -> no human progress by default
```

Why:

- The ingestion pipeline knows what phase and page it is on.
- The CLI knows whether the caller is a human terminal or a JSON consumer.
- Keeping events presentation-agnostic makes the behavior testable without terminal snapshots.

Alternative considered:

- Print directly from `rules.py`. Rejected because it would mix domain work with terminal rendering and risks corrupting JSON output.

### 2. Render progress to stderr and keep final JSON on stdout

Progress and live status belong on stderr. Final JSON output must remain on stdout exactly as before when `--json` is used.

Why:

- Agents and scripts can keep parsing `--json` output deterministically.
- Human terminal users still see progress while work runs.
- stderr is the normal place for interactive progress indicators.

Alternative considered:

- Add progress fields to JSON as streaming stdout lines. Rejected for this change because it would create a different protocol and complicate existing JSON consumers.

### 2a. Use automatic progress by default for the first slice

The first implementation slice should not add a `--progress` option. Progress should be visible by default for non-JSON human runs:

- interactive terminal: render Rich live progress/status on stderr
- non-interactive, non-JSON human output: print plain phase/status lines
- JSON mode: keep stdout parseable and suppress human progress unless a future explicit streaming mode is designed

Why:

- The core UX bug is that a normal human invocation looks dead.
- Adding `auto`, `always`, and `never` modes before the first visible-progress pass creates option surface before there is evidence it is needed.
- Plain phase lines keep redirected logs and CI-like non-JSON runs understandable without relying on terminal control sequences.

Alternative considered:

- Add `--progress auto|always|never` immediately. Deferred because automatic human-mode progress solves the current problem with less interface complexity.
- Keep non-interactive human output quiet until completion. Rejected because redirected or captured non-JSON runs would still look silent during long ingests.

### 3. First visible output should happen before per-page extraction

Human mode should print a start banner or status line after basic validation and PDF inspection, before the command enters the per-page extraction loop. The banner should include the book identity, source PDF, selected page count or page range, and target rules database.

Example shape:

```text
Ingesting Core Rulebook
Book:   core_rules_v5 (core)
Source: ~/Downloads/v5_core_rules.pdf
Pages:  431
Target: .backet/rules/rules.sqlite3
```

Why:

- A large PDF can make the command look dead if nothing appears before the first slow phase.
- The user can immediately confirm that the right book, source, and vault target were chosen.

Alternative considered:

- Wait until extraction starts to show progress. Rejected because PDF open/inspection and initial setup can still feel like a silent hang.

### 4. Treat extraction and OCR as the main progress surface

Page extraction should show page-based progress. OCR fallback should be explicitly visible because OCR pages explain most surprising slowdowns. Counters should distinguish:

- pages processed
- pages requiring OCR
- pages still needing review or low-confidence follow-up
- chunks created when that count is available

Why:

- A clean extracted page and an OCR page have very different runtime costs.
- The user needs to know whether slowness is normal OCR work or a stalled command.
- Suspect pages are an action signal; OCR pages are context.

Alternative considered:

- A single spinner for the whole ingest. Rejected because it proves liveness but does not explain slow phases or give useful scale.

### 5. Use a custom human completion report for rules ingest

`backet rules ingest` should use a command-specific human report instead of the generic `emit_success()` dump. The `CommandResult` should remain complete for JSON, but human output should:

- omit empty fields
- summarize long page lists
- label OCR pages as pages that required OCR
- label suspect pages as pages needing review
- show counts before details
- provide `backet rules audit` and targeted repair guidance when review is needed

Example shape:

```text
Ingested Core Rulebook

Book:    core_rules_v5 (core)
Pages:   431 processed
Chunks:  1,169 stored
OCR:     62 pages required OCR
Review:  24 pages need review

Stored:  .backet/rules/rules.sqlite3
Source:  ~/Downloads/v5_core_rules.pdf

Review recommended:
  Pages needing review: 34, 43, 48, 52, 64, 68, 73, 76, 79, 85, +14 more
  Run: backet rules audit . --book-id core_rules_v5
```

Why:

- Generic data dumps bury the important outcome.
- Humans need interpretation: OCR is not necessarily bad, but suspect pages require attention.
- Long arrays should be available in JSON, not forced into the default human report.

Alternative considered:

- Improve the generic `emit_success()` renderer. Rejected because only command-specific context can label `suspect_pages` and `ocr_used_on_pages` correctly.

## Risks / Trade-offs

- [Progress events add coupling between CLI and rules ingestion] -> Keep event names small, domain-oriented, and presentation-agnostic.
- [Terminal progress can make tests brittle] -> Unit test event emission and report text separately from full Rich rendering.
- [stderr progress could surprise scripts that do not use `--json`] -> Keep default progress human-oriented and document that scripts should use `--json` for stable machine output.
- [Timing assertions are fragile] -> Specify first output before long-running per-page extraction rather than relying only on wall-clock timing tests.
- [Long page lists still matter for troubleshooting] -> Preserve full lists in JSON and summarize only the human report.

## Migration Plan

1. Add a progress event type or callback protocol inside the rules ingestion module.
2. Emit events around PDF inspection, per-page extraction, OCR fallback, storage/chunking, search index rebuild, and audit summary.
3. Add a CLI progress reporter for human `rules ingest` runs, writing progress/status to stderr.
4. Add a command-specific human completion report for rules ingestion.
5. Preserve existing JSON result fields and tests for `--json`.

Rollback is straightforward: disable the CLI reporter and fall back to the existing `CommandResult` emission. The persisted rules corpus does not change.

## Open Questions

- None for the first slice. Automatic human-mode progress is enough, and non-interactive non-JSON output should print plain phase lines.
